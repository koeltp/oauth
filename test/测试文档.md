# OAuth 服务端测试文档

## 概述

本文档覆盖 OAuth 服务端全部测试场景，包含三个测试项目：

| 项目 | 类型 | 协议 | 运行端口 | 典型场景 |
|---|---|---|---|---|
| TestClient.Web | Confidential Client | Authorization Code + PKCE | 5002 | 有后端的 Web 应用 |
| TestClient.Spa | Public Client | Authorization Code + PKCE | 5003 | 纯前端单页应用 / 移动端 |
| TestClient.Console | Confidential Client | Client Credentials | 命令行 | 微服务 / 定时任务 |

## 快速运行

```bash
# 从仓库根目录执行

# Web 测试应用 — 浏览器访问 http://localhost:5002
dotnet run --project TestClient.Web

# 控制台测试套件 — 交互式命令行菜单
dotnet run --project TestClient.Console

# SPA 测试应用 — 浏览器访问 http://localhost:5003
node TestClient.Spa/server.js
```

> **前置条件：** 确保 OAuth 服务端已启动，测试客户端已在服务端注册并审核通过。

## TestClient.Web —— 测试用例（11 项）

### 前置条件

- 服务端已启动并运行
- 已在服务端注册测试客户端
- `appsettings.json` 配置正确

### 1.1 标准登录（PKCE）—— `/login`

| 项目 | 内容 |
|---|---|
| 测试目的 | 验证完整的 Authorization Code + PKCE 流程 |
| 前置步骤 | 无 |
| 测试步骤 | 1. 访问首页，点击「标准登录（PKCE）」 |
| | 2. 浏览器跳转到 OAuth 授权页，确认授权 |
| | 3. 回调页面显示 Token 和 UserInfo |
| 期望结果 | Token 换取成功，UserInfo 返回用户信息 |
| 验证点 | code_verifier 浏览器生成 → SHA-256 → code_challenge → 服务端验证通过 |

### 1.2 标准登录（无 PKCE）—— `/no-pkce`

| 项目 | 内容 |
|---|---|
| 测试目的 | 验证 Authorization Code 流程（仅靠 client_secret 保护） |
| 前置步骤 | 无 |
| 测试步骤 | 1. 访问首页，点击「标准登录（无 PKCE）」 |
| | 2. 浏览器跳转到 OAuth 授权页，确认授权 |
| | 3. 回调页面显示 Token 和 UserInfo |
| 期望结果 | Token 换取成功 |
| 验证点 | 无 code_challenge 参数时，服务端跳过 PKCE 校验 |

### 1.3 自定义 Scope —— `/test-scopes`

| 项目 | 内容 |
|---|---|
| 测试目的 | 验证不同的 scope 参数对 Token 和 UserInfo 返回结果的影响 |
| 前置步骤 | 无 |

| | 2. 分别选择不同的 scope 组合发起登录 |

| 期望结果 | scope=openid → 仅返回 sub |
| | scope=openid profile → 返回 sub + name + 头像等 |
| | scope=openid email → 返回 sub + email |

### 1.4 Code 重复使用 —— `/code-reuse-test`

| 项目 | 内容 |
|---|---|
| 测试目的 | 验证服务端是否正确标记 authorization code 为已使用 |
| 前置步骤 | 先完成一次标准登录（PKCE 或无 PKCE） |
| 测试步骤 | 1. 登录成功后，点击「Code 重复使用测试」 |

| 期望结果 | 服务端返回 `invalid_grant`，拒绝重复使用 |
| 安全要求 | 必须拒绝，防止重放攻击 |

### 1.7 错误 redirect_uri —— `/wrong-redirect-uri`

| 项目 | 内容 |
|---|---|

### 1.10 Refresh Token 旋转 —— `/refresh-rotation-test`

| 项目 | 内容 |
|---|---|
| 测试目的 | 验证 Refresh Token 旋转机制（旧 token 刷新后被吊销） |
| 前置步骤 | 1. 先完成标准登录获取 Token |
| | 2. 点击回调页面的「刷新 Token」，获取新 token 并吊销旧 token |
| 测试步骤 | 1. 上一步刷新后，访问 `/refresh-rotation-test` |

| 安全要求 | 必须拒绝，Refresh Token 旋转是 OAuth 安全最佳实践 |

## 安全漏洞修复验证

### 漏洞 #1：redirect_uri 未校验

**修复内容：**
- 授权请求中的 `redirect_uri` 被存储到 Authorization 记录
- Token 换取时必须传入相同的 `redirect_uri`
- 不一致则返回 `invalid_grant`

**验证方式：**
```
TestClient.Web → /wrong-redirect-uri
```
用 `redirect_uri = http://localhost:9999/fake-callback` 换取 Token

**预期结果：**
```json
{
  "error": "invalid_grant",
  "error_description": "redirect_uri mismatch"
}
```

### 漏洞 #2：PKCE code_verifier 未验证

**修复内容：**
- 授权请求带 `code_challenge` 时，Token 请求必须带 `code_verifier`
- 服务端对 code_verifier 做 SHA-256 哈希后与 code_challenge 比对
- 支持 `S256` 和 `plain` 两种模式

**验证方式：**
```
TestClient.Web → /wrong-code-verifier
```
用随机生成的 fake code_verifier 换取 Token

**预期结果：**
```json
{
  "error": "invalid_grant",
  "error_description": "code_verifier mismatch"
}
```

## 客户端类型对比

| 特性 | Confidential Client (Web) | Public Client (SPA) | Confidential Client (Console) |
|---|---|---|---|
| 有 client_secret | ✅ | ❌ | ✅ |
| PKCE 支持 | ✅ 可选 | ✅ 强制 | 不适用 |
| Grant Type | authorization_code | authorization_code | client_credentials |
| 用户交互 | ✅ 浏览器 | ✅ 浏览器 | ❌ 命令行 |
| 典型实现 | ASP.NET Core / Spring Boot | React / Vue / 移动端 | 微服务 / CLI 工具 |
| 安全机制 | client_secret + PKCE | PKCE | client_secret |
